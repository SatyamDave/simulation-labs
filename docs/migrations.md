# Database migrations (Alembic)

Simulation Labs / Ghostpanel uses [Alembic](https://alembic.sqlalchemy.org/) to
version the SQL schema. The schema of record is the SQLModel metadata in
`src/ghostpanel/store/models.py`; Alembic turns changes to those models into
ordered, reviewable migration scripts under `migrations/versions/`.

> **Production rule.** In production the schema MUST be managed by
> `alembic upgrade head`. `SQLModel.metadata.create_all` (`store.db.init_db`) is
> **dev/test only** — it cannot alter existing tables, drop columns, or record a
> version, so it silently drifts. Prod never calls `create_all`.

## How it's wired

- **`alembic.ini`** — config. `script_location = migrations`, and
  `sqlalchemy.url` is deliberately **blank**.
- **`migrations/env.py`** — resolves the database URL at runtime from
  `ghostpanel.server.config.get_settings().effective_database_url` (which honours
  the `DATABASE_URL` env var), and points autogenerate at
  `ghostpanel.store.models`'s `SQLModel.metadata`.

### Async vs sync: we run migrations synchronously

The app runs on an **async** engine (`sqlite+aiosqlite` in dev,
`postgresql+asyncpg` in prod). Alembic's DDL and schema inspection are
synchronous, so `env.py` converts the app's async URL to its **sync driver
equivalent** and runs on a plain sync engine — no event loop, works identically
for SQLite and Postgres:

| App (async) URL              | Migration (sync) URL          | Extra dependency        |
|------------------------------|-------------------------------|-------------------------|
| `sqlite+aiosqlite:///...`    | `sqlite:///...`               | none (stdlib)           |
| `postgresql+asyncpg://...`   | `postgresql+psycopg://...`    | `psycopg[binary]` (v3)  |
| `mysql+aiomysql://...`       | `mysql+pymysql://...`         | `PyMySQL`               |

For SQLite, `env.py` enables Alembic **batch mode** (`render_as_batch=True`) so
`ALTER TABLE` operations work (SQLite's ALTER is limited); it's a no-op on
Postgres. `compare_type=True` is on so column-type changes are detected on
autogenerate.

> **Postgres prod note:** install a sync Postgres driver alongside `asyncpg`.
> `psycopg` (v3) is recommended: `uv pip install "psycopg[binary]"`. No app code
> changes — only migrations use it.

## Everyday commands

Run from the repo root with the venv active (`source .venv/bin/activate`). All
commands read `DATABASE_URL` if set, else fall back to the dev SQLite file
(`ghostpanel.db`).

### Create a new migration after changing models

```bash
alembic revision --autogenerate -m "add widgets table"
```

Then **review the generated file** in `migrations/versions/`. Autogenerate is a
draft — it does not reliably detect renames (sees drop+add), server-side
defaults, or some constraint changes. Edit before committing.

### Apply migrations

```bash
alembic upgrade head        # apply everything up to the latest
alembic upgrade +1          # apply the next one only
```

### Roll back

```bash
alembic downgrade -1        # undo the most recent migration
alembic downgrade base      # undo everything (empty schema)
```

### Inspect

```bash
alembic current             # revision the DB is at
alembic history --verbose   # full revision graph
alembic heads               # tip revision(s)
```

### Offline / SQL-only (generate DDL without touching a DB)

```bash
alembic upgrade head --sql > migration.sql
```

## Targeting a specific / throwaway database

`DATABASE_URL` is an async URL (env.py converts it):

```bash
# a scratch SQLite file — never the repo's real ghostpanel.db
DATABASE_URL="sqlite+aiosqlite:///$TMPDIR/scratch.db" alembic upgrade head

# production Postgres
DATABASE_URL="postgresql+asyncpg://user:pw@host/db" alembic upgrade head
```

## Deploy checklist

1. Merge model + migration changes together (never a model change without its
   migration).
2. On deploy, run `alembic upgrade head` **before** starting the app processes.
3. The app must not call `init_db` / `create_all` in prod (dev-only helper).

## Baseline

The first revision, `0b2fc46c1a86 initial schema`, creates the full current
schema: `users`, `projects`, `memberships`, `api_keys`, `runs`, `baselines`,
`jobs` (with their indexes, foreign keys, and unique constraints). It was
autogenerated from the SQLModel metadata and verified to produce a table set
identical to `create_all` on a fresh SQLite database.
