Overview

Simulation Labs builds synthetic users that do, not say. Instead of asking a language model to opine about your checkout flow, we point a swarm of synthetic users at a live website with a concrete goal (sign up, check out, cancel) and watch each one either finish or abandon at a specific recorded pixel. Failure is the product.

The engine (internal package ghostpanel) drives a screenshot-in / action-out vision model — H Company's Holo (holo3-1-35b-a3b, a pre-trained model we consume; we do not train it) — around a real browser via Playwright. Because Holo sees pixels and emits clicks, we control both channels and mechanically degrade them to model real perceptual and motor constraints. We do not claim psychological fidelity — we claim mechanical fidelity: the constraints are real even if the personality is synthetic. Prompted LLM personas match real users only about 11.86% of the time (arXiv 2503.20749); the argument here is to degrade the channel, not to prompt a personality.

Status. Working end-to-end — five backend modules plus a React frontend, verified live. The test suite has 173 tests. MIT-licensed. Built at H Company's Computer Use Hackathon, San Francisco, Jul 2026 (Track 2 — Browser Use).

Architecture

The system is a five-module frozen-contract design. Every message that crosses a module boundary is a Pydantic v2 model or a @runtime_checkable Protocol defined in shared/ghostpanel_contracts/contracts.py — never a bare dict or tuple. That file is frozen (CONTRACT_VERSION = "1.0.0"); each module can assert against the version to detect a stale checkout. The design let five parallel agents build on five branches under a no-overlap file-ownership map, so the branches merged without conflicts.

  • engine — perturbation transforms, the Live/Fake Holo clients, the persona agent, and the persona loader.
  • runner — PlaywrightSessionRunner: the per-session loop, video capture, thumbnails, stuck/success detection, and network policy.
  • server — a FastAPI orchestrator that acts as the composition root, wiring the concrete classes together, fanning out over personas with asyncio.gather, and running a WebSocket hub plus the runs/leaderboard API.
  • report — SurvivalReportBuilder: survival curve, abandonment heatmap, the composite score and per-run statistics, and a standalone HTML report.
  • voice — GradiumVoiceEngine: cloned-voice exit interviews with optional Claude narration polish (the default is a template; narration is not always on).

The composition root builds one Chromium browser and shares it; each persona runs as its own asyncio task with its own isolated browser context. Live progress is pushed to the frontend over a WebSocket as a discriminated union of RunEvent messages (run_started, persona_started, step, persona_finished, run_finished), so the grid stays alive step by step.

The run loop

One session drives a single persona from the target URL to a terminal outcome. Each step captures a screenshot, hands a perturbed copy to Holo, jitters the returned coordinates, and executes the action verbatim, emitting a StepEvent (thumbnail + human caption) every step (src/ghostpanel/runner/session.py). 

for step in range(persona.max_steps):
    png    = page.screenshot()                 # true viewport, un-perturbed
    frame  = perceive(png, persona)             # downscale -> blur -> CVD (dims preserved)
    action = holo.navigate(frame, task, hist)   # Holo returns an Action
    action = actuate(action, persona, w, h)     # add tremor jitter, clamp to viewport
    emit StepEvent(caption, thumbnail, x, y)    # live grid tile

    if action is ANSWER and success_predicate holds:  outcome = SUCCESS;      break
    if is_stuck(history):                              outcome = STUCK;        break
    execute_action(page, action); settle()             # real page time charged to sim-clock
    if success_predicate holds:                        outcome = SUCCESS;      break
    if sim_seconds >= persona.deadline_s:              outcome = TIME_BUDGET;  break
else:
    outcome = STEP_BUDGET                               # ran out of steps

A session ends in exactly one of five outcomes: success (a claimed completion is verified against a success predicate — "receipts, not vibes"), step_budget, time_budget, stuck (a loop / no-progress detector tripped), or error (infra failure — excluded from survival statistics, never a persona verdict). Time is measured on a simulated persona clock — page loads, waits, actions, and a fixed per-step think cost — which excludes Holo inference latency and rate-limiter queueing, so the time verdict is rate-limit-independent.

Perturbation channels

Perturbations are pure, deterministic functions (src/ghostpanel/engine/perturbation.py). Perception transforms run in the order downscale → blur → CVD. Tremor is an actuation transform applied to coordinates after Holo decides.

ChannelMechanismParameter
Blur PIL GaussianBlur over the frame before Holo sees it blur_sigma (radius, px)
Downscale Resize down by a factor then back up in place — drops high-frequency detail downscale_factor (0–1)
Colour DaltonLens Machado-2009 simulator (deutan / protan / tritan) cvd_type, cvd_severity (0–1)
Tremor Add N(0, σ) Gaussian noise to click (x, y), then round and clamp tremor_sigma_px
Small viewport A tiny Playwright viewport (typically with tremor) viewport (e.g. 390×844)
Impatience Hard step ceiling plus a wall-clock deadline on the sim-clock max_steps, deadline_s
Literal reading A note appended to the navigation prompt constraining to literal, low-inference reading literacy_note

The golden rule: every perception transform is dimension-preserving. Holo tokenizes at the image's dimensions (smart-resize), so keeping dimensions constant means the coordinates it returns never have to be remapped. Blur and CVD change pixel values only; downscale resizes down then back up in place. The pipeline asserts img.size == original_size before returning.

Coordinate space

The hosted Holo 3.1 API returns coordinates normalized to a 0–1000 grid, not absolute pixels. LiveHoloClient always denormalizes them against the image it sent (x / 1000 × w, y / 1000 × h) and clamps to the viewport, so downstream code receives true viewport pixels and the runner executes them verbatim with page.mouse.click(x, y). (FakeHoloClient returns pixel coords directly and does not denormalize.) 

# Verified live: a button whose true centre is (547, 430) in a 1280x800 shot
Holo returns   Click(426, 536)        # 0-1000 normalized grid
denormalize    426/1000 * 1280 = 545  # x, rounded
               536/1000 *  800 = 429  # y, rounded
runner clicks  (545, 429)             # true viewport pixels, back on target

Inference runs at temperature 0. A single shared asyncio token-bucket rate limiter paces the whole swarm to one RPM budget (the free tier is 10 RPM). The parser is tolerant of malformed model output: it tries JSON first, repairs Holo's known key-dropped coordinate shapes (e.g. {"action": "click", "x": 426, 536} with a missing "y" key), then falls back to a Click(x, y) short form and finally a keyword or center-click salvage so the run keeps making progress.

Behavioral models

Persona physics are mechanical, not roleplayed. Each model is a JSON file (personas/*.json) validated against PersonaConfig. The two control models apply no perturbation at all.

idBehaviourMechanicsSteps / time
power-user Fluent, patient baseline — the control for comparison No perturbation 40 / 240s
ai-agent Headless agent — is the site usable by an agent at all? No perturbation 40 / 240s
low-vision Can't read small or low-contrast text blur σ=3.5 + downscale ×0.4 25 / 150s
tremor Clicks land off-target and miss tiny controls tremor σ=14px 30 / 150s
colorblind Colour-only cues and error states vanish deutan severity 0.9 30 / 120s
impatient-mobile In a hurry on a small phone; abandons the moment a flow feels slow 390×844 viewport + tremor σ=6 + literal prompt 8 / 30s
grandma-72 Reads every label literally, mistrusts icons, gives up quickly blur σ=1.2 + tremor σ=4 + literal prompt 12 / 90s
non-native Idioms and jargon slow them down literal prompt + impatience (language it) 20 / 100s

Outputs & scoring

A finished run produces a stepped survival curve (how many personas are still alive at each step), an abandonment heatmap (where sessions died, in viewport pixels), and per-persona statistics: average and 95th-percentile Holo latency, blocked_actions, and max_repeated_action — the longest run of consecutive identical captions, our "rage-click" metric (src/ghostpanel/report/insights.py).

The composite Simulation Score (ghostpanel_score in the JSON) is 0–100, with equal weight per non-error persona: 

credit(persona) = 1.0                                  if outcome == SUCCESS
                = min(0.5, 0.5 * steps_survived / max_steps)   otherwise

score = round(100 * sum(credit) / count(non-error personas))

A success earns full credit; a failure earns partial credit for distance travelled, capped at 0.5 — so dying at step 2 drags the score down more than dying at step 25. A separate agent-readiness score is derived from the unimpaired ai-agent control (100 on success; the same partial credit otherwise): if an unimpaired agent can't finish, the site isn't agent-ready.

We also emit a WCAG "evidence of risk" table that maps each persona's active perturbation channels to WCAG 2.2 success criteria (with the corresponding EN 301 549 clause). Each entry is grounded in the action trace — the failure step, the failure pixel, and the last action. This is evidence of risk, not an automated conformance verdict and not a legal certification.

Policy & safety

Runs are browse-only. A YAML request policy (a NemoClaw mirror) is enforced in-process at the Playwright network layer: every browser request is intercepted, and anything the policy denies — including form POSTs — is aborted before it leaves the machine (fail-closed: a broken policy engine blocks rather than bypasses). A blocked request is recorded against the step and surfaced as a shielded StepEvent in the live grid.

As a result we do not complete real purchases or submissions on third-party sites; a persona that claims it is "done" after a blocked submission is not scored as a success. The full NVIDIA OpenShell / Docker sandbox and on-device Nemotron are out of scope — they need a GPU — so on a laptop the policy is the in-process network gate described above, not the full sandbox.

Contracts reference

These are the key models in shared/ghostpanel_contracts/contracts.py. Everything crossing a module boundary is one of them. 

PersonaConfig    a synthetic user: perturbation params, budgets, literacy note
Viewport         width x height for the browser context
Observation      one frame handed to the agent (raw un-perturbed PNG + metadata)
Action           a decoded Holo action in TRUE viewport pixels (+ caption, raw)
ActionType       click | write | scroll | go_back | refresh | wait | goto | restart | answer
PerturbationKind blur | downscale | cvd | tremor | small_viewport | impatience | low_literacy
PersonaOutcome   success | step_budget | time_budget | stuck | error
StepRecord       one step: action + thumbnail + latency + note
PersonaResult    a full session: outcome, steps, failure coords/step/reason, video, transcript
SurvivalPoint    one persona's survival summary for the curve
HeatPoint        an (x, y, weight) point for the abandonment heatmap
RunReport        the aggregate: results + survival + heatmap + completion rate
RunEvent         discriminated union on `event` (the exact WebSocket JSON):
                   RunStarted | PersonaStarted | StepEvent | PersonaFinished | RunFinished
