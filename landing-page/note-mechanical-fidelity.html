Mechanical fidelity: degrading perception and action instead of prompting personas

Satyam Dave

Technical note · Jul 2026 · Simulation Labs

Abstract. Synthetic-user tools ask a language model to describe how a persona would behave. We instead make an agent attempt the task on a live site and mechanically degrade the perceptual and motor channels it acts through. Because our model, H Company's Holo (holo3-1-35b-a3b), is screenshot-in / action-out, we can transform exactly the pixels it sees and the clicks it emits without touching the model itself. We claim mechanical fidelity, not psychological fidelity: the constraints on the channel are real even though the personality is synthetic. This note describes the motivation, the approach, the perturbation channels, and the coordinate handling that keeps clicks exact.

1 Introduction and motivation

There is a gap between what users say they would do and what they actually do. Interview-style synthetic users sit on the wrong side of that gap: they generate plausible opinions about a checkout flow rather than trying to complete it. Recent work reports that prompted LLM personas reproduce real user behavior only about 11.86% of the time (arXiv 2503.20749), and usability practitioners have noted that synthetic-user answers tend to be shallow and people-pleasing (Nielsen Norman Group).

Our reading of that evidence is that the failure is structural, not a prompting problem. Asking a model to role-play a personality and then predict its behavior compounds two sources of error. So we do not prompt a personality. We take a capable agent and constrain the channel it operates through — what it can perceive, and how precisely it can act — so that failure emerges from the mechanics of the interaction rather than from a narrative the model invents. The output is not an opinion; a persona either completes the task or abandons at a specific recorded pixel. Failure is the product.

2 Approach

Holo is a screenshot-in / action-out model: it receives an image of the page and returns a single UI action (click, type, scroll, and so on). This is the property we exploit. We insert two deterministic transforms into the loop — one on the way in and one on the way out — and leave the model untouched. We consume the pre-trained model; we do not train or fine-tune it.

The runner loop is: take a Playwright screenshot, run perceive() to perturb the frame, send it to Holo, receive an action, run actuate() to apply motor noise, then execute the action verbatim with page.mouse.click(x, y) and emit a step event with a thumbnail and a caption. This repeats until success, a step budget, a wall-clock budget, or a stuck-detector ends the session.

The one rule that makes this safe is what we call the golden rule: every image transform is dimension-preserving. Holo tokenizes an image at its own dimensions, so if we never change the width and height, the coordinates it returns never need to be remapped. Blur and colour transforms change pixel values only; the low-acuity transform resizes down and then back up to the original size in place. The pipeline asserts img.size == original_size before returning.

3 Perturbation channels

Each persona enables some subset of the channels below. All the image channels are dimension-preserving; tremor and the budget channels act on the action and the loop rather than the frame.

Blur (low acuity)PIL GaussianBlur(radius=σ) applied to the frame before Holo sees itYes
Downscale (low acuity)Resize down by a factor, then back up to the original size (loses high-frequency detail)Yes
Colour (CVD)DaltonLens Machado-2009 simulator; deutan / protan / tritan, severity 0–1Yes

ImpatienceHard max_steps plus a wall-clock deadline_s (sim-clock excludes model queue time)n/a (loop)

Agent (control)No perturbation — answers "is the site usable by an agent at all?"Yes (identity)

The perception transforms compose in a fixed order: downscale, then blur, then CVD. We describe personas in behavioral terms (for example, "can't read small or low-contrast text" or "clicks land off-target") rather than as diagnoses; the mechanics above are the method, not a claim about any individual.

4 Coordinate handling

The hosted Holo3.1 API does not return absolute pixels. It returns coordinates normalized to a 0–1000 grid. LiveHoloClient denormalizes them against the dimensions of the image it sent — x/1000·w, y/1000·h — and clamps to the viewport, so everything downstream of the engine receives true viewport pixels and executes them verbatim.

We verified this against a live call. A button whose true centre was at (547, 430) in a 1280×800 screenshot came back from the API as Click(426, 536); denormalizing gives (545, 429), within rounding of the true target. This is also why the golden rule matters: if a perturbation changed the frame's dimensions, the denormalization target would no longer be the real viewport and every click would drift.

5 Limitations

We claim mechanical fidelity, not psychological fidelity. Degrading a channel reproduces a constraint on perception or motor control; it does not reproduce a person's goals, patience, mental model, or emotional state. A persona is a control model, not a human being.

Our live evidence is thin. The end-to-end system runs and we have observed real completions and abandonments, but the runs that use the live model have very small n; the large-n results we report elsewhere are offline or synthetic. We use a single model family (Holo3.1) at temperature 0, so we cannot yet separate what the channel does from what this particular model does. Treat the outputs as rank-order signals about where a flow is worse or whether a fix helped, not as forecasts of real-world abandonment.
