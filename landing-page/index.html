three steps

From flow to failure report.

  1. 01

    Define your flow

    Give us a URL and a goal — sign up, check out, cancel — in one line.

  2. 02

    Deploy behavioral personas

    A swarm of computer-use agents attempts the task the way real users would.

  3. 03

    Get the failure report

    See the exact pixel where each persona quit, and why, with receipts.

the output

Receipts, not opinions.

Every run turns into hard evidence — and every run rolls up into a dashboard that tracks the flow release over release.

Per-run reports

  • Survival curve Who made it, and how far each one got.
  • Abandonment heatmap Where users die on your actual page.
  • Video receipts The recording of every failure, step by step.
  • Cloned-voice exit interviews Each persona explains why it gave up, out loud.

Analytics dashboard the platform layer

  • Completion rate over releases Is the flow getting better or worse?
  • Failure points aggregated The drop-offs that show up run after run.
  • Persona-segmented breakdown Which behaviors fail, and where.
  • Before/after comparison Prove the fix moved the number.

who it's for

One engine, three teams.

cro / growth

Optimize flows you already run

Tune the flows that already carry intent — checkout, signup, subscription upgrade. Run before/after tests on every redesign and watch completion move in the dashboard over time.

product launch

Validate what you're about to ship

Check a new onboarding flow, dashboard, or feature before launch. There's no user data yet, so a simulation is often your only pre-launch signal — and the moment you most want reassurance.

qa / engineering

natural expansion

Test every deploy automatically

Plug into CI/CD and run the personas against every build before it ships — a pre-deploy check, not a manual pass. The build fails when users would.

powered by

The engine under the hood.

Real tools doing real work — here's exactly what each one does.

  • H Company Holo core engine

    The core computer-use engine (holo3-1-35b-a3b). Every persona sees a screenshot and returns a real click.

  • Gradium

    Cloned per-persona voices for the spoken exit interviews.

  • Anthropic Claude

    Optional narration polish for exit interviews.1

  • NVIDIA NemoClaw

    A network-layer policy that keeps runs read-only — live form POSTs are blocked before they leave the machine.2

1 Claude is optional and narration-only. Without it, a grounded template is used — it is never central to the result.

2 On a laptop the same browse-only YAML policy is enforced in-process at the browser network layer; the full GPU sandbox is out of scope for the demo.
